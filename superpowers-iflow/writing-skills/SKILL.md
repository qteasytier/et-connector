# 编写技能

## 概述

**编写技能是应用于过程文档的测试驱动开发。**

**个人技能驻留在特定于代理的目录中（Claude Code 为 `~/.claude/skills`，Codex 为 `~/.codex/skills`）**

你编写测试用例（带子代理的压力场景），看着它们失败（基线行为），编写技能（文档），看着测试通过（代理遵守），并重构（关闭漏洞）。

**核心原则：** 如果你没有看着代理在没有技能的情况下失败，你就不知道技能是否教对了东西。

**必需背景：** 在使用此技能之前，你必须理解 superpowers:test-driven-development。该技能定义了基本的红-绿-重构循环。此技能将 TDD 适应于文档。

## 什么是技能？

**技能**是经过验证的技术、模式或工具的参考指南。技能帮助未来的 Claude 实例找到并应用有效的方法。

**技能是：** 可重用的技术、模式、工具、参考指南

**技能不是：** 关于你如何解决一次问题的叙述

## 技能的 TDD 映射

| TDD 概念 | 技能创建 |
|----------|----------|
| **测试用例** | 带子代理的压力场景 |
| **生产代码** | 技能文档（SKILL.md） |
| **测试失败（红）** | 代理在没有技能的情况下违反规则（基线） |
| **测试通过（绿）** | 代理在技能存在时遵守 |
| **重构** | 在保持合规的同时关闭漏洞 |
| **先写测试** | 在编写技能之前运行基线场景 |
| **看着它失败** | 记录代理使用的确切合理化 |
| **最小代码** | 编写针对那些特定违规的技能 |
| **看着它通过** | 验证代理现在遵守 |
| **重构循环** | 找到新的合理化 → 堵塞 → 重新验证 |

整个技能创建过程遵循红-绿-重构。

## 何时创建技能

**创建当：**
- 技术对你来说不直观
- 你会在项目中再次引用它
- 模式广泛适用（不是项目特定的）
- 其他人会受益

**不要创建用于：**
- 一次性解决方案
- 其他地方有良好文档的标准做法
- 项目特定约定（放在 CLAUDE.md 中）
- 机械约束（如果可以通过正则/验证强制执行，自动化它 - 为判断调用保存文档）

## 技能类型

### 技术
有步骤要遵循的具体方法（条件基于等待、根本原因跟踪）

### 模式
思考问题的方式（用标志展平、测试不变量）

### 参考
API 文档、语法指南、工具文档（办公室文档）

## SKILL.md 结构

**Frontmatter（YAML）：**
- 只支持两个字段：`name` 和 `description`
- 最多 1024 个字符
- `name`：仅使用字母、数字和连字符（没有括号、特殊字符）
- `description`：第三人称，仅描述何时使用（不描述它做什么）
  - 以"Use when..."开始以专注于触发条件
  - 包括具体症状、情况和上下文
  - **从不总结技能的过程或工作流**（见 CSO 部分解释原因）
  - 如果可能，保持在 500 字符以下

## 关键原则

1. **测试先行** - 在编写技能之前运行基线场景
2. **验证合规** - 在部署前用技能重新测试场景
3. **最小化** - 只写解决基线失败的内容
4. **迭代** - 发现新漏洞 → 修复 → 重新测试

## 铁律（与 TDD 相同）

```
没有失败测试就没有技能
```

这适用于新技能和现有技能的编辑。

**没有例外：**
- 不是用于"简单添加"
- 不是用于"只是添加一节"
- 不是用于"文档更新"
- 不要保留未测试的更改作为"参考"
- 不要在运行测试时"适应"
- 删除意味着删除

## 红旗

- 技能前测试
- 实施后测试
- 测试立即通过
- 无法解释为什么测试失败
- 稍后添加测试
- 合理化"只是这次"
- "我已经手动测试了它"
- "测试后达到相同目的"
- "是关于精神而不是仪式"
- "保留作为参考"或"适应现有代码"
- "已经花费 X 小时，删除是浪费的"
- "TDD 是教条的，我是务实的"
- "这不同因为..."

**所有这些都意味着：删除代码。从 TDD 重新开始。"

## 底线

**创建技能是应用于过程文档的 TDD。**

相同的铁律：没有失败测试就没有技能。
相同的循环：红（基线）→ 绿（编写技能）→ 重构（关闭漏洞）。
相同的好处：更好的质量，更少的惊喜，防弹结果。

如果你对代码遵循 TDD，对技能也遵循它。这是应用于文档的相同纪律。
