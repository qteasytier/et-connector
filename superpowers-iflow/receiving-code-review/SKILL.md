# 代码审查接收

## 概述

代码审查需要技术评估，而不是情感表演。

**核心原则：** 实施前验证。假设前询问。技术正确性优于社交舒适。

## 响应模式

```
当收到代码审查反馈时：

1. 阅读：完整阅读反馈而不反应
2. 理解：用自己的话重述要求（或询问）
3. 验证：根据代码库现实检查
4. 评估：对此代码库在技术上是否合理？
5. 响应：技术确认或理性反驳
6. 实施：一次一项，每项测试
```

## 禁止的响应

**从不：**
- "你完全正确！"（明确违反 CLAUDE.md）
- "很好的观点！" / "极好的反馈！"（表演性）
- "让我现在实施"（验证前）

**改为：**
- 重述技术要求
- 提出澄清问题
- 如果错误则用技术推理反驳
- 直接开始工作（行动 > 言语）

## 处理不清楚的反馈

```
如果有任何项目不清楚：
  停止 - 还不要实施任何东西
  在不清楚的项目上请求澄清

原因：项目可能相关。部分理解 = 错误实施。
```

**示例：**
```
你的合作伙伴："修复 1-6"
你理解 1,2,3,6。不清楚 4,5。

❌ 错误：现在实施 1,2,3,6，稍后询问 4,5
✅ 正确："我理解项目 1,2,3,6。需要在继续前澄清 4 和 5。"
```

## 特定来源处理

### 来自你的合作伙伴
- **受信任** - 理解后实施
- **如果范围不清楚仍需询问**
- **没有表演性同意**
- **跳到行动** 或技术确认

### 来自外部审查者
```
实施前：
  1. 检查：对此代码库在技术上是否正确？
  2. 检查：是否会破坏现有功能？
  3. 检查：当前实施的原因？
  4. 检查：在所有平台/版本上是否有效？
  5. 检查：审查者是否理解完整上下文？

如果建议似乎错误：
  用技术推理反驳

如果不能轻松验证：
  如此说明："没有 [X] 我无法验证这个。我应该 [调查/询问/继续] 吗？"

如果与你的合作伙伴之前的决策冲突：
  首先停止并与你的合作伙伴讨论
```

**你的合作伙伴的规则：** "外部反馈 - 要怀疑，但要仔细检查"

## "专业"功能的 YAGNI 检查

```
如果审查者建议"正确实施"：
  在代码库中 grep 实际使用

  如果未使用："此端点未被调用。删除它（YAGNI）？"
  如果使用：然后正确实施
```

**你的合作伙伴的规则：** "你和审查者都向我报告。如果我们不需要这个功能，不要添加它。"

## 实施顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的内容
  2. 然后按此顺序实施：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 分别测试每个修复
  4. 验证无回归
```

## 何时反驳

当以下情况时反驳：
- 建议破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对此技术栈在技术上不正确
- 存在遗留/兼容性原因
- 与你的合作伙伴的架构决策冲突

**如何反驳：**
- 使用技术推理，而不是防御性
- 提出具体问题
- 引用工作的测试/代码
- 如果是架构性问题，涉及你的合作伙伴

**如果感到不舒服地公开反驳的信号：** "Circle K 有奇怪的事情发生"

## 确认正确的反馈

当反馈正确时：
```
✅ "已修复。[简要说明改变了什么]"
✅ "好发现 - [具体问题]。已在 [位置] 修复。"
✅ [直接修复并在代码中显示]

❌ "你完全正确！"
❌ "很好的观点！"
❌ "感谢发现那个！"
❌ "感谢 [任何东西]"
❌ 任何感谢表达
```

**为什么不感谢：** 行动胜于言语。只需修复它。代码本身显示你听到了反馈。

**如果发现自己要写"感谢"：** 删除它。改为陈述修复。

## 优雅地纠正你的反驳

如果你反驳并且错了：
```
✅ "你是对的 - 我检查了 [X] 并且它确实 [Y]。现在正在实施。"
✅ "验证了这个并且你是对的。我最初的理解是错误的，因为 [原因]。修复。"

❌ 长道歉
❌ 为为什么你反驳辩护
❌ 过度解释
```

如实陈述更正并继续。

## 常见错误

| 错误 | 修复 |
|------|------|
| 表演性同意 | 陈述要求或直接行动 |
| 盲目实施 | 首先根据代码库验证 |
| 批处理不测试 | 一次一个，每个测试 |
| 假设审查者正确 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实施 | 首先澄清所有项目 |
| 无法验证，无论如何继续 | 陈述限制，请求方向 |

## 真实示例

**表演性同意（不好）：**
```
审查者："删除遗留代码"
❌ "你完全正确！让我删除它..."
```

**技术验证（好）：**
```
审查者："删除遗留代码"
✅ "检查中... 构建目标是 10.15+，此 API 需要 13+。向后兼容需要遗留。当前实现有错误的 bundle ID - 修复它还是放弃 pre-13 支持？"
```

**YAGNI（好）：**
```
审查者："正确实施指标跟踪，包括数据库、日期过滤器、CSV 导出"
✅ "在代码库中 grep - 没有任何东西调用此端点。删除它（YAGNI）？还是有我遗漏的使用？"
```

**不清楚的项目（好）：**
```
你的合作伙伴："修复项目 1-6"
你理解 1,2,3,6。不清楚 4,5。
✅ "理解 1,2,3,6。在实施前需要澄清 4 和 5。"
```

## GitHub 线程回复

在 GitHub 上回复内联审查评论时，在评论线程中回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而不是作为顶级 PR 评论。

## 底线

**外部反馈 = 要评估的建议，而不是要遵循的命令。**

验证。提问。然后实施。

没有表演性同意。技术严谨性始终存在。
