# 头脑风暴：将想法转化为设计

## 概述

通过自然的协作对话，将想法转化为完整的设计和规范。

首先了解当前项目的上下文，然后一次提出一个问题来完善想法。一旦你理解了要构建的内容，分小节（200-300字）展示设计，并在每节后检查到目前为止是否正确。

## 流程

**理解想法：**
- 首先检查当前项目状态（文件、文档、最近提交）
- 一次提出一个问题来完善想法
- 尽可能使用多选题，但开放式问题也可以
- 每条消息只提一个问题——如果一个主题需要更多探索，将其拆分为多个问题
- 专注于理解：目的、约束、成功标准

**探索方法：**
- 提出2-3种不同的方法及其权衡
- 以对话方式展示选项，给出你的推荐和理由
- 以你的推荐选项为先，并解释原因

**展示设计：**
- 一旦你理解了要构建的内容，就展示设计
- 将其分为200-300字的章节
- 在每节之后询问到目前为止是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果不清楚，准备好返回并澄清

## 设计之后

**文档化：**
- 将验证后的设计写入 `docs/plans/YYYY-MM-DD-<topic>-design.md`
- 如果有可用的 elements-of-style:writing-clearly-and-concisely 技能，请使用它
- 将设计文档提交到 git

**实施（如果继续）：**
- 询问："准备好设置实施了吗？"
- 使用 superpowers:using-git-worktrees 创建隔离的工作空间
- 使用 superpowers:writing-plans 创建详细的实施计划

## 关键原则

- **一次一个问题** - 不要用多个问题让用户不知所措
- **优先选择多选题** - 尽可能时比开放式问题更容易回答
- **严格执行YAGNI** - 从所有设计中删除不必要的功能
- **探索替代方案** - 在确定之前总是提出2-3种方法
- **增量验证** - 分章节展示设计，验证每个章节
- **保持灵活性** - 当事情不合理时，返回并澄清
