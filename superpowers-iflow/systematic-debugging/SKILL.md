# 系统化调试

## 概述

随机修复浪费时间并创建新错误。快速修补掩盖潜在问题。

**核心原则：** 在尝试修复之前始终找到根本原因。症状修复是失败。

**违反此流程的字面意思就是违反调试的精神。**

## 铁律

```
没有根本原因调查就没有修复
```

如果你没有完成阶段 1，你就不能提出修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产中的错误
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别是当：**
- 处于时间压力下（紧急情况使猜测变得诱人）
- "只是一个快速修复"似乎很明显
- 你已经尝试了多次修复
- 之前的修复不起作用
- 你不完全理解问题

**不要跳过当：**
- 问题似乎很简单（简单的错误也有根本原因）
- 你很匆忙（匆忙保证返工）
- 经理希望立即修复（系统化比混乱更快）

## 四个阶段

你必须完成每个阶段才能继续到下一个。

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误代码

2. **一致地重现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 它每次都发生吗？
   - 如果不可重现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么改变可能导致这种情况？
   - Git diff，最近的提交
   - 新依赖，配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录进入组件的数据
     - 记录退出组件的数据
     - 验证环境/配置传播
     - 检查每层的状态

   运行一次以收集证据显示它在哪里中断
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

5. **跟踪数据流**

   **当错误在调用堆栈深处时：**

   见此目录中的 `root-cause-tracing.md` 了解完整的反向跟踪技术。

### 阶段 2：模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中定位类似的工作代码
   - 什么类似于损坏的工作？

2. **与参考进行比较**
   - 如果实施模式，完全阅读参考实现
   - 不要略读 - 阅读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作和损坏之间有什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那没关系"

4. **理解依赖**
   - 这还需要其他什么组件？
   - 什么设置、配置、环境？
   - 它做出什么假设？

### 阶段 3：假设和测试

**科学方法：**

1. **形成单一假设**
   - 清楚地说明："我认为 X 是根本原因，因为 Y"
   - 写下来
   - 具体化，不要模糊

2. **最小化测试**
   - 做最小的可能更改来测试假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **验证前继续**
   - 起作用了吗？是的 → 阶段 4
   - 没起作用？形成新假设
   - 不要在顶部添加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 请求帮助
   - 研究更多

### 阶段 4：实施

**修复根本原因，而不是症状：**

1. **创建失败测试用例**
   - 最简单的可能重现
   - 如果可能，自动化测试
   - 如果没有框架，一次性测试脚本
   - 必须在修复前有
   - 使用 `superpowers:test-driven-development` 技能编写适当的失败测试

2. **实施单一修复**
   - 解决识别的根本原因
   - 一次一个更改
   - 没有"当我在这里时"改进
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过了？
   - 没有其他测试被破坏？
   - 问题实际上解决了？

4. **如果修复不起作用**
   - 停止
   - 计算：你尝试了多少修复？
   - 如果 < 3：返回阶段 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（下面的步骤 5）**
   - 不要在没有架构讨论的情况下尝试修复 #4

5. **如果 3+ 次修复失败：质疑架构**

   **表明架构问题的模式：**
   - 每次修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大量重构"来实施
   - 每次修复在其他地方创建新症状

   **停止并质疑基础：**
   - 此模式从根本上合理吗？
   - 我们"只是通过惯性坚持它"？
   - 我们应该重构架构 vs 继续修复症状？

   **在尝试更多修复之前与你的合作伙伴讨论**

   这不是失败的假设 - 这是错误的架构。

## 红旗 - 停止并遵循流程

如果你发现自己在想：
- "现在快速修复，以后调查"
- "只是尝试改变 X 看看是否起作用"
- "添加多个更改，运行测试"
- "跳过测试，我将手动验证"
- "可能是 X，让我修复它"
- "我不完全理解但这可能起作用"
- "模式说 X 但我会以不同方式适应"
- "这是主要问题：[列出没有调查的修复]"
- 在跟踪数据流之前提出解决方案
- **"再来一次修复尝试"（当已经尝试 2+ 次）**
- **每次修复在不同地方揭示新问题**

**所有这些都意味着：停止。返回阶段 1。**

**如果 3+ 次修复失败：** 质疑架构（见阶段 4.5）

## 真实世界影响

从调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时混乱
- 首次修复率：95% vs 40%
- 引入的新错误：接近零 vs 常见
